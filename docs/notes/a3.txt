The best version of your idea is **an airlock, not a mirror**.

Do **not** try to store the whole world locally. Make the source of truth a **signed global approval manifest**, then put a tiny **pull-through verifier/cache** in front of Git and package fetches. When a developer or CI asks for code, the edge node checks the manifest. If the repo/ref/version is approved and the fetched code hashes to the approved digest, it serves it. If it is unknown, it goes to quarantine, deception, and your waiting window. That model is much simpler to operate, much easier to self-host, and lines up with recent supply-chain failures where mutable tags were part of the problem. In the March 2025 `tj-actions/changed-files` compromise, tags were retroactively moved to a malicious commit, and GitHub’s own guidance says pinning a **full-length commit SHA** is the only immutable way to reference an action. ([GitHub][1])

My top pick for the project name is **Airlock**.

The product map would be:

* **Airlock Index**: the signed public or private approval manifest
* **Airlock Edge**: the tiny internal pull-through verifier/cache
* **Frostline-14**: the waiting period
* **Mirage Lab**: the deception-heavy honeypot
* **Mirage Feed**: the hourly public hits site and enterprise alert stream

## What I would build

### 1. Airlock Index: a signed global manifest, not a giant flat JSON

Use **TUF** for manifest distribution. TUF gives you signed metadata, expiration, versioning, Snapshot protection against mix-and-match, and delegated targets so you can shard the manifest instead of shipping one monster file. The required top-level roles are Root, Targets, Snapshot, and Timestamp, and TUF metadata is meant to expire so clients can reject stale state. ([TUF][2])

The manifest should approve **immutable content**, not friendly names. That means every approval record is keyed by a canonical source identity plus an exact resolved object:

```text
source: git:https://github.com/org/repo
selector: refs/tags/v1.4.2
resolved_commit: <git oid>
tree_oid: <git tree oid>
treeseal_sha256: <your canonical tree hash>
submodules: [path, url, commit, treeseal_sha256]
lfs_objects: [path, sha256]
status: approved | pending | rejected | superseded
first_seen_at:
approved_at:
hold_until:
risk:
attestations:
```

The smart move is to store **both** the upstream Git identity and **your own SHA-256 content hash**. Git does have a SHA-256 repository format now, but older Git versions cannot read SHA-256 repos, so you should not force the ecosystem to migrate just to use Airlock. Keep compatibility by recording the upstream Git object IDs and adding your own canonical SHA-256 digest over the checked-out tree. ([Git][3])

I would call that canonical digest **TreeSeal**. Do **not** hash raw packfiles; pack transport is not the trust boundary. Hash the normalized file tree instead.

### 2. TreeSeal: the hash that actually matters

For Git content, hash the unpacked, normalized tree, not the transport blob. TreeSeal v1 can be:

* paths sorted lexicographically
* for each entry: mode + path + NUL + sha256(content)
* symlinks hash target bytes
* submodules hash the recorded commit plus their own approved TreeSeal
* Git LFS pointers are resolved and actual object digests recorded separately

That gives you a stable trust hash even if packfile ordering or compression changes.

Also recurse through submodules. Git records the submodule commit ID in the superproject, so they are part of what you are approving. ([Git][4])

### 3. Airlock Edge: the smallest possible developer change

This is the clever trick: **use Git’s built-in URL rewriting**.

Git supports `url.<base>.insteadOf`, and the rewrite happens before Git talks to the remote. Git also documents that the **longest matching** `insteadOf` wins, and `git ls-remote --get-url` will expand the rewritten URL without contacting the remote. Git config can be installed system-wide via `/etc/gitconfig`. ([Git][5])

So the smallest rollout is not “change every repo.” It is **zero repo changes** plus one machine or runner bootstrap:

```ini
# /etc/gitconfig or managed user config
[url "https://edge.company.internal/gh/"]
    insteadOf = https://github.com/
    insteadOf = ssh://git@github.com/
    insteadOf = git@github.com:

[url "https://edge.company.internal/gl/"]
    insteadOf = https://gitlab.com/
    insteadOf = ssh://git@gitlab.com/
    insteadOf = git@gitlab.com:
```

That means code can still say `https://github.com/org/repo`, but Git actually fetches from your internal node.

One important caveat: **config alone is not enforcement**. Git documents `GIT_CONFIG_NOSYSTEM`, which disables system config, so real enforcement also needs egress policy. Developer and CI networks should be unable to reach public Git hosts directly except from the Airlock puller tier. ([Git][6])

### 4. What Airlock Edge does on every request

Flow:

```text
Developer/CI -> Git URL rewrite -> Airlock Edge
   -> check signed manifest
      -> approved?
         -> yes: fetch exact ref/commit into clean pull sandbox
              -> compute TreeSeal
              -> match manifest?
                 -> yes: cache read-only and serve
                 -> no: reject + alert
         -> no: queue to Mirage Lab + Frostline-14
              -> return structured fallback
```

This is the crucial simplification: **Airlock Edge is a pull-through verifier with a bounded cache**, not a full mirror. It only stores things that somebody actually asked for.

I would make the fallback policy:

* If the request is a **floating selector** like a branch or a semver-style selector, Airlock can automatically serve the **latest approved** version that satisfies policy.
* If the request is an **exact commit or exact tag** that is not approved, **do not silently downgrade in CI**. Return a clear block message plus a machine-readable recommendation for the nearest approved version.

That gives developers a fast answer without turning exact pins into hidden behavior changes.

Example response from the resolve API:

```json
{
  "status": "pending",
  "reason": "Frostline-14",
  "requested": "github.com/acme/lib#v2.6.1",
  "approved_fallback": "v2.5.9",
  "approved_commit": "4d9c...",
  "hold_until": "2026-04-15T00:00:00Z"
}
```

Interactive tooling can auto-accept that fallback in dev mode. CI should fail closed.

### 5. Provenance: treat upstream attestations as signals, then add your own

Use **Cosign** for signing and verifying blobs and artifacts, and represent your own evidence as **in-toto attestations** with **SLSA provenance**. Both GitHub artifact attestations and PyPI digital attestations now exist, so Airlock should ingest upstream provenance when available, but never treat it as sufficient by itself. Your system should generate a fresh internal attestation for fetch, hash, sandbox verdict, hold-until, and promotion. OCI distribution is content-type agnostic, so it is a good backing store for signed evidence bundles and other non-container artifacts too. ([Sigstore][7])

## Mirage Lab: the honeypot

This is where your idea gets genuinely special.

The mistake would be to run suspicious code in an ordinary Docker container on a normal host. The better design is:

**container payload inside a sealed microVM**

I would use **Firecracker** as the outer boundary, then run a rootless container runtime inside the guest. Firecracker is designed for secure, multi-tenant microVM execution with a minimal device model, and Docker’s rootless mode exists specifically to reduce daemon/runtime privilege exposure. Add `no_new_privs`, Landlock, seccomp, and aggressive capability dropping. Docker’s default seccomp profile is only a moderate baseline, so tighten it. ([GitHub][8])

I would structure Mirage Lab around **personas**, not one static fake machine:

* **CI runner**: fake GitHub/GitLab env vars, secrets, checkout, artifact tokens
* **Developer workstation**: fake SSH keys, browser-ish state, `.npmrc`, `.pypirc`, `.cargo`, `.netrc`, `.env`
* **Cloud workload**: fake kubeconfig, fake service account files, fake IMDS, fake Vault, fake S3
* **Container builder**: fake registry creds, fake Docker config, fake build context
* **High-value decoy**: intentionally richer bait for “grab everything” malware

Every secret is synthetic and uniquely watermarked per run.

### Fake internet, no real internet

Let malicious code think it can phone home, but keep it inside a **sinkhole network**:

* wildcard DNS to your fake resolver
* fake HTTP/HTTPS endpoints
* fake metadata endpoints
* fake Git hosting
* fake webhook receivers
* fake package indexes
* fake Docker API
* fake Kubernetes API
* fake Redis/Postgres/Vault

There is **no route to the outside world** from the microVM. The code can try; every attempt is captured.

Never expose a real Docker socket as bait. Docker documents that the `docker` group grants root-level privileges. If you want that lure, expose a **fake** Docker API that only records access and returns believable results. ([Docker Documentation][9])

### Sensors

Use overlapping layers:

* **Tracee** for broad eBPF event capture
* **Falco** for near-real-time rules
* **Tetragon** for selective in-kernel enforcement on truly bad behaviors

Tracee exposes hundreds of system/runtime events via eBPF, Falco is designed for near-real-time runtime threat detection with rules, and Tetragon can enforce by overriding a function return or sending `SIGKILL`. ([Aqua Security][10])

Tripwires should include:

* fake secret file reads
* `.ssh`, `.aws`, `.kube`, `.npmrc`, `.pypirc`, `.cargo/credentials`, `.docker/config.json`
* shell spawns during install/build
* downloader behavior
* new executable creation
* `/proc` inspection
* unexpected child processes
* outbound DNS and connection attempts
* metadata service hits
* fake Docker/K8s/Vault/IMDS access
* attempts to escape namespace or mount boundaries

### Frostline-14

Unknown or changed content moves into:

`DISCOVERED -> PENDING -> MIRAGE_OK -> FROSTLINE_14 -> APPROVED`

Reset the timer if:

* a tag moves
* the same version resolves to a different commit
* signer identity changes
* install/build hooks appear
* submodule map changes
* native code appears where none existed
* any Mirage rule hits

This matters because recent supply-chain attacks have exploited mutable refs and surprise behavior changes. ([GitHub][1])

## The security log and public site

The public hourly website should not dump your entire lure strategy. It should publish:

* package/repo
* requested selector
* resolved commit
* status: pending / rejected / approved
* first seen
* detector family
* severity
* safe fallback version
* ecosystem
* signed evidence bundle ID

Enterprise customers get the full forensic view:

* process tree
* file timeline
* bait touched
* DNS and network attempts
* syscall highlights
* fake secret IDs
* PCAP pointers
* screenshot/video if desired
* exact rule hits
* replay history across personas

Make the UI timeline-first. Security people want to answer: **what did it touch, what did it try to reach, and when did it become untrustworthy?**

## Beyond Git

Git is the universal wedge, but the same model extends to registries cleanly. Cargo supports **source replacement** for mirroring, npm can use any compatible registry, pip explicitly warns that `--extra-index-url` is unsafe because of dependency confusion, Maven supports repository mirrors, Go obeys `GOPROXY`, and NuGet’s Package Source Mapping exists to constrain which source a package can come from. ([Rust Documentation][11])

So phase 1 is Git and GitHub Actions. Phase 2 is registry facades.

For GitHub Actions specifically, mirror third-party actions internally and allow only **full commit SHA** references. GitHub says that is the only immutable release form. ([GitHub Docs][12])

## Rust-first implementation

I would make the control plane strongly Rust:

* **gix / gitoxide** for Git parsing, resolution, and hashing
* **axum** for HTTP APIs
* **tokio** for async orchestration
* **sqlx** for metadata storage
* Rust CLI for bootstrap, policy, and replay

`gitoxide` is a pure Rust Git implementation, `axum` is a Rust web framework, `tokio` is the async runtime, and `sqlx` is a pure Rust async SQL toolkit. ([GitHub][13])

I would keep one repo, one Cargo workspace, many deliverables:

```text
airlock/
  crates/
    airlock-index/
    airlock-edge/
    airlock-lab/
    airlock-treeseal/
    airlock-policy/
    airlock-feed/
    airlock-site/
    airlock-cli/
  policies/
  personas/
  rules/
  deploy/
  docs/
```

Then publish separate artifacts:

* `airlock-edge`
* `airlock-lab`
* `airlock-full-stack`

That gives you one source tree but lets people download **just the honeypot**.

## Open source vs enterprise

Open-source should include:

* manifest format
* client and edge proxy
* TreeSeal hashing
* basic Mirage Lab
* public hit feed
* self-host docs
* watchlist mode
* long-list batch mode

Enterprise can add:

* private overlay manifest
* proprietary personas and bait packs
* bigger rule library
* live push alerts
* private IOC feed
* higher-frequency refresh
* private-source scanning
* compliance and audit exports

The key is that enterprise features should be **additive**, not a fork of the protocol.

## My strongest recommendation

Do **not** market this as “game over for all supply attacks.” That is too strong.

Market it as:

**“No direct internet code. Only approved, attested, hashed, sandboxed, held-back code may enter the build.”**

That is a real promise.
